## 二、技术决策变更

### 变更 1: ASR 语音识别 — 七牛云 → 阿里云百炼

| | 计划 | 实际 |
|------|------|------|
| 服务商 | 七牛云短语音听写 | 阿里云百炼 |
| 模型 | — | `paraformer-realtime-v2` |
| 认证 | HMAC-SHA1 签名 (AppID + SecretKey) | Bearer Token (单一 API Key) |
| 实现 | `backend/asr/qiniu_asr.py` | `backend/asr/paraformer_asr.py` |

**原因**: 七牛云 ASR 接口复杂（双 Key + HMAC 签名），阿里云百炼 REST API 更简洁（Bearer Token 直传）。`paraformer-realtime-v2` 支持普通话 + 粤语，同步模式直接返回结果，无需轮询。

**潜在优化**: 后续可评估 Qwen3-Audio-ASR 或 Fun-ASR 等百炼平台更新模型（本次因联网搜索不可用未对比，先保留 `paraformer-realtime-v2`）。

---

### 变更 2: LLM 大模型 — 七牛云 MaaS → DeepSeek 官方直连

| | 计划 | 实际 |
|------|------|------|
| 渠道 | 七牛云 MaaS 代理 (`api.qnaigc.com`) | DeepSeek 官方 API (`api.deepseek.com`) |
| 模型 | `deepseek-v3` | `deepseek-v4-flash` |
| API Key | 七牛云下发 | DeepSeek 平台直接获取 |

**原因**: 
1. **减少中间层** — 七牛云 MaaS 代理增加了一个依赖方，直连更稳定
2. **模型升级** — DeepSeek 官方公告 `deepseek-v3` / `deepseek-chat` 将于 2026/07/24 停用，提前迁移到 V4 系列
3. **性价比** — `deepseek-v4-flash` 速度快、成本低，绘图指令生成场景够用；需要更强推理时可切换 `deepseek-v4-pro`

---

### 变更 3: 默认模型降级策略

| | 计划 | 实际 |
|------|------|------|
| LLM 默认 | `deepseek-v3` | `deepseek-v4-flash` |
| ASR 降级 | ASR 失败 → Web Speech | 同（未变） |
| ASR 不配 | 不可用 | 允许（前端自动降级浏览器语音识别） |

**新增能力**: `.env` 不配 `DASHSCOPE_API_KEY` 也能跑 —— 前端检测 ASR 错误后自动切换浏览器内置 Web Speech API。最少只需一个 DeepSeek Key 即可跑通全链路。

## 六、经验总结

### 做得好的

1. **PR 拆分粒度合理** — 每个 PR 独立可验收，依赖链清晰，并行度高
2. **前后端分离彻底** — 通过 JSON 指令协议解耦，两边独立调试
3. **降级策略到位** — ASR 失败 → Web Speech API，LLM 失败 → 友好提示，单点故障不崩
4. **安全校验层** — Command Parser 7 项校验在 LLM 输出后兜底，哪怕 Prompt 被绕过也不会执行危险指令

### 可改进的

1. **入口文件遗漏** — PR #10 合入后 `main.tsx` 没更新导入 App，导致白屏。应在 PR checklist 中加一项"入口文件同步检查"
2. **API 选型可前置** — 七牛云 ASR 和 MaaS 在实际接入时才发现不够好用，提前做 PoC 验证可避免返工
3. **UI 打磨时间预留不足** — 2.5 天冲刺中 UI 被挤到最后，Demo 效果受影响
